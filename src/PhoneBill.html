Phone Bill Manager

Java REST API Android OOP Client–Server TDD / JUnit Maven Servlet / HTTP File I/O Javadoc / JaCoCo

A full-stack phone billing system built across five iterative project phases — evolving from a command-line Java app into a networked client–server system with an Android front-end. The application models real-world billing infrastructure: customers can be created, phone calls logged against their account, and records queried or filtered by date-range.

The server exposes a RESTful HTTP API backed by a custom Java Servlet. The Android client communicates with this server via HTTP GET and POST requests, while also maintaining a local file-based data store for offline persistence. On startup, users can push locally saved bills back to the server — giving the app genuine dual-layer data resilience.

Built with a strict test-driven development workflow: JUnit tests were written before implementation, coverage tracked with JaCoCo, and builds automated end-to-end through Maven.

Key Implementation Details

REST Servlet (GET & POST)

PhoneBillServlet handles GET requests to retrieve a full bill or search calls between two timestamps, and POST requests to add new calls. Customer bills are held in a HashMap<String, PhoneBill> on the server for fast lookup by customer name.

Date-Range Filtering

The search endpoint parses timestamps with SimpleDateFormat and compares Date.compareTo() to include only calls whose start and end fall within the queried window — mirroring production billing query logic.

Dual-Layer Persistence

The Android client writes call data to a local billList.txt via TextDumper, then on relaunch offers to sync that file back to the server — intentionally decoupling local state from server state.

Sorted Call List

PhoneBill.sorted() runs a bubble-sort over the internal PhoneCall[] array after every insertion, normalizing phone number formats (dashed, spaced, plain) before numeric comparison to guarantee consistent ordering regardless of input style.

Code Samples

The servlet's doPost() extracts and validates all five required parameters before constructing a PhoneCall, then either creates a new PhoneBill or appends to an existing one in the server map. Validate first, mutate state only on success — a defensive pattern used in production APIs.

PhoneBillServlet.java  —  doPost() handler

String customer = getParameter("customer", request);
if (customer == null) { missingRequiredParameter(response, "customer"); return; }
// ... validate caller, callee, start, end ...

PhoneCall tcall = new PhoneCall(customer, tcaller, tcallee, start, end);
PhoneBill bill1 = this.phoneBillMap.get(customer);

if (bill1 == null) {
  bill1 = new PhoneBill(tcall);          // first call: create bill
  this.phoneBillMap.put(customer, bill1);
} else {
  bill1.addPhoneCall(tcall);             // subsequent: append + sort
  this.phoneBillMap.put(customer, bill1);
}
response.setStatus(HttpServletResponse.SC_OK);

PhoneBillRestClient.java  —  date-range search

public int getSearch(String customer, String start, String end)
    throws IOException, ParserException {

  Response response = http.get(Map.of(
    "customer", customer,
    "start",    start,    // MM/dd/yyyy hh:mm a
    "end",      end
  ));

  if (response.getHttpStatusCode() != HTTP_OK) {
    System.err.println("No calls found in range");
    return 0;
  }
  System.out.println(response.getContent());
  return 1;
}

PhoneBillRestClient wraps HttpRequestHelper into three typed, intention-revealing methods: add (POST), retrieve by name (GET), and search by range (GET with three params). This keeps Project4.java free of raw HTTP plumbing, and surfaces non-200 responses as readable error messages rather than raw exceptions.
