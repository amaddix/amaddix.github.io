CIFAR-10 image recognition algorithm comparison

Python Language - Machine Learning - Object Detection - performance optimization - framework proficiency - AI algorithm comparison research - analytical mindset

REPORT PRESENTATION




Team Research project : Myself along with, Forest Pearson, Ian Wasson, Eben Weisman, Jingjing Zhao


Overview In this research-intensive project, I spearheaded a comparative analysis of supervised and unsupervised learning paradigms to solve a high-dimensional computer vision problem. Using the CIFAR-10 dataset—a benchmark collection of 60,000 color images—I developed, optimized, and benchmarked three distinct algorithmic approaches: a Multi-Layer Perceptron (MLP), a Convolutional Neural Network (CNN), and a K-Means Clustering algorithm.

Phase 1: Supervised Learning & Architecture Optimization The project began with the development of a custom Multi-Layer Perceptron (MLP). This phase focused on the mathematical foundations of deep learning, where I implemented manual backpropagation and weight-update logic. Through rigorous hyperparameter tuning, I analyzed the relationship between model complexity and convergence. By experimenting with hidden unit density and momentum values, I discovered that a leaner architecture (50 units) paired with lower momentum ($0.5$) actually mitigated oscillations and improved accuracy from a baseline of 43% to 60%. This demonstrated my ability to use empirical data to refine model performance rather than relying on default configurations.

Phase 2: High-Performance Computing with CNNs To push the boundaries of the dataset, I transitioned to a Convolutional Neural Network (CNN) built with TensorFlow and Keras. Recognizing the massive computational overhead of processing 32x32 RGB tensors, I integrated NVIDIA CUDA and cuDNN to leverage GPU acceleration. This architectural shift was transformative: Efficiency: Training time per epoch dropped to just 6 seconds, a massive improvement over the 12-hour training cycles required for the unoptimized MLP. Spatial Intelligence: By utilizing convolutional layers that reference neighboring pixel data, the model achieved a peak testing accuracy of 70%+. Analytical Insight: I identified and documented significant overfitting (94% training vs. 68% testing), providing a foundation for future implementation of dropout layers and data augmentation.

Phase 3: Unsupervised Learning & Dimensionality Finally, I implemented a K-Means Clustering algorithm to evaluate how unsupervised models handle high-dimensional raw image data. The primary challenge was the "Curse of Dimensionality"—transforming 3,072-dimensional image vectors into meaningful clusters. By optimizing matrix formations and centroid convergence logic, I achieved a stable classification rate of 23%. While expectedly lower than the supervised models, this provided critical insights into the inherent structure of the CIFAR-10 data and the limitations of heuristic clustering for complex visual tasks.

Key Technical Competencies Demonstrated Frameworks & Tools: TensorFlow, Keras, NumPy, Matplotlib. Deep Learning: CNN Architecture, Backpropagation, Hyperparameter Optimization, Gradient Descent. Hardware Acceleration: GPU Computing (NVIDIA CUDA/cuDNN). Data Science: Dimensionality Reduction, Unsupervised Clustering, Statistical Analysis of Model Loss.

Conclusion: This project demonstrates my proficiency in the full machine learning lifecycle: from data preprocessing and custom algorithm implementation to hardware-level optimization and performance benchmarking. I possess the technical agility to switch between high-level frameworks like TensorFlow and low-level mathematical implementations, ensuring I can both build and explain complex software systems.





Pizza Builder

Linked Lists OOP / Classes Dynamic Memory Recursion Sorted Insertion Pointer Arithmetic File Architecture Encapsulation Defensive Programming


This project is an interactive, menu-driven C++ application for building and managing a custom pizza catalog. Users can create named pizza types, add and remove toppings with detailed metadata (name, category, surcharge), and display their full menu — all through a console interface built entirely without standard library containers.

The core data structure is a two-level nested linked list — an outer sorted list of pizza types, each owning its own inner sorted list of toppings. Every node is heap-allocated using raw pointers and manual memory management, and the destructor carefully traverses both levels to fully clean up without leaks.

Key Implementation Details

Sorted Insertion

Both pizza types and toppings are inserted alphabetically on the fly using strcmp. All three edge cases — head, middle, and tail — are handled explicitly without a separate sort pass.

Recursive Display

The public display_pizza(int) wrapper calls a private recursive overload — a classic abstraction pattern demonstrating function overloading and intentional interface design.

Destructor Cleanup

The destructor traverses both list levels — deallocating each topping's inner char* fields before releasing outer nodes — a disciplined, leak-free teardown of complex nested heap structures.

find_pizza() Reuse

A private helper sets an internal current pointer on match and is shared across add_topping, remove_topping, and display_topping — no duplicated traversal logic.

Each struct is designed with a clear purpose — pizza_type holds a name, a pointer to its own inner topping list, and a next pointer to chain with other pizza types. pizza_topping mirrors this pattern one level down. Together they form the two-level nested structure that the entire program is built around.

  head = temp;  head->next = NULL;  return 1;

All three cases are handled explicitly — an empty list, a new name that sorts before the current head, and a name that belongs somewhere in the middle — using a previous/current pointer traversal to find the right position and splice the node in.

Message Manager

Stack (LLL of Arrays) Circular Linked List Queue OOP / Classes Dynamic Memory Recursion Pointer Arithmetic Encapsulation Randomized Dispatch


This project is a menu-driven C++ messaging simulation that models how an inbox and outbox work at the data structure level — with no standard library containers. Received messages are stored in a stack built from a linked list of arrays, while outgoing messages queue up in a circular linked list and are dispatched in randomized batches using rand().

Every field in every message is heap-allocated on input and the destructors for both classes fully traverse their respective structures, deallocating each char* field and node without leaks. Reply messages carry a reference to the original message they respond to, making the outbox context-aware.

Key Implementation Details

Stack — LLL of Arrays

The inbox is not a simple stack — each node holds a fixed-size array of MAX messages. When an array fills, a new node is allocated and pushed to the front. This hybrid structure demonstrates understanding of both contiguous and linked memory models simultaneously.

Queue — Circular Linked List

The outbox is implemented as a circular linked list where queue_rear->next always points back to the head. This allows O(1) enqueue at the tail and O(1) dequeue from the head — with careful pointer management to convert between CLL and linear form for counting.

Recursive Count

Because the queue is circular, traversal requires temporarily breaking the ring. The public count() wrapper converts the CLL to a linear list, calls the private recursive overload, then restores the ring — separating traversal concerns cleanly and protecting the data structure's integrity.

Context-Aware Replies

Each q_node carries both a reply and a sent message field plus a response bool. When a user replies, the original incoming message is saved alongside the outgoing one — modelling real-world message threading at the struct level.

Code Samples

The first snippet is the destructor for the circular queue. Because queue_rear->next always points back to the head rather than NULL, a standard traversal loop would never terminate. The solution is to sever that back-link first, converting the circular structure into a regular linear list, then traverse and deallocate normally.

queue.cpp  —  ~replied_message() destructor

if (queue_rear) {
  // break the ring: save head, sever rear->next
  q_node *head = queue_rear->next;
  queue_rear->next = NULL;
  queue_rear = head;

  while (queue_rear) {
    clear_reply();
    if (queue_rear->responce) clear_sent();
    head = head->next;
    delete queue_rear;
    queue_rear = head;        // advance — terminates at NULL
  }
  queue_rear = NULL;
}

queue.cpp  —  count() wrapper + recursive overload

// Public wrapper: temporarily linearise the CLL, count, restore ring
int replied_message::count() {
  if (!queue_rear) return 0;
  if (queue_rear->next == queue_rear) return 1;

  q_node *head      = queue_rear->next;
  q_node *rear_temp = queue_rear;
  queue_rear->next  = NULL;          // sever: CLL → LLL
  queue_rear        = head;

  int amount = count(queue_rear);    // recurse on linear list

  queue_rear        = rear_temp;     // restore rear pointer
  queue_rear->next  = head;          // re-link: LLL → CLL
  return amount;
}

// Private recursive overload
int replied_message::count(q_node *queue_rear) {
  if (!queue_rear) return 0;
  return count(queue_rear->next) + 1;
}

The second snippet shows the count() function, which faces the same challenge. The public wrapper temporarily breaks the ring, saves the rear pointer, then calls the private recursive overload on the now-linear list. Once counting is done, the ring is fully restored before returning. This keeps the recursive logic simple while protecting the structure's circular integrity.

CS Glossary — Hash Table

Hash Table Separate Chaining File I/O OOP / Classes Dynamic Memory Recursion Pointer Arithmetic Encapsulation Bulk Source Removal


This project is a menu-driven C++ glossary application that stores and retrieves computer science definitions using a hash table with separate chaining. Terms can be loaded in bulk from a pipe-delimited text file or added one at a time by the user. Each entry stores a term name, definition, source, and citation — all heap-allocated — and is inserted at the head of the linked list at its hashed index.

The hash function converts a term into a table index by summing its ASCII character values and taking the result modulo the table size. Collisions are handled by chaining nodes in a linked list at each index, meaning multiple terms can share a slot without overwriting each other. The destructor traverses every chain in the array, deallocating each char* field and node fully before freeing the array itself.

Key Implementation Details

Hash Function

The index is computed by summing the ASCII values of every character in the key and taking the result modulo the table size (61). This distributes terms across the array based on their content, giving O(1) average-case lookup without any sorting overhead.

Separate Chaining

Each index of the array holds the head of a singly linked list. Collisions are resolved by prepending new nodes to the front of the chain — a clean O(1) insert that intentionally preserves all entries rather than overwriting them, which also allows a term to exist under multiple sources.

Bulk Source Removal

remove_all_source() scans every chain in the entire table and deletes any node whose ref field matches the given source. It handles both mid-chain removal using a previous/current pointer pair and head removal as a separate edge case, returning the total count of deleted entries.

Recursive Display

The public display_all() iterates over the array and calls a private recursive helper display_all_recursion() on each chain. The helper prints one node, recurses on next, and accumulates the count — keeping the loop and the traversal logic cleanly separated.

Code Samples

The first snippet shows the hash function and add() together. The hash function sums ASCII values and mods by the table size to produce an index, then add() allocates a new node and prepends it to the front of the chain at that index. This pair shows how the two responsibilities — computing where a term goes and physically inserting it — are kept in separate functions while working together cleanly.

table.cpp  —  hash_function() + add()

// Sum ASCII values of each character, mod by table size
int table::hash_function(char *key_value) {
  int sum = 0;
  for (int i = 0; key_value[i] != '\0'; ++i)
    sum += key_value[i];
  return sum %= table_size;
}

// Prepend new node to the front of the chain at the hashed index
int table::add(char *key_value, definition users) {
  int index = hash_function(key_value);
  node *temp = new node;
  temp->next        = hash_table[index]; // point new node to current head
  hash_table[index] = temp;              // new node becomes new head
  copy(users, index);
  return 1;
}

Within each chain it uses a previous/current pointer pair to unlink mid-chain nodes, then checks the head separately as an edge case. This demonstrates the difference between a targeted hash lookup and a full-table scan, and why each requires a different traversal strategy.

CS Glossary — Binary Search Tree

Binary Search Tree BST Deletion (3 Cases) In-Order Traversal Recursion OOP / Classes Dynamic Memory File I/O Wrapper/Recursive Pattern Encapsulation


This project is a menu-driven C++ glossary application that organizes computer science definitions in a Binary Search Tree — keeping terms in sorted alphabetical order at all times without any external sorting step. Each node stores a term, definition, source, and citation — all heap-allocated — and the tree is maintained purely through pointer manipulation with no standard library containers.

Every public operation follows a clean wrapper/recursive pattern: the public method accepts user-facing arguments, then delegates to a private recursive function that traverses the tree by value comparison. The destructor uses post-order recursion to fully deallocate every node and its fields without leaks, and the tree height can be queried at any time for structural insight.

Key Implementation Details

BST Insert

Insertion is handled recursively using a reference-to-pointer parameter (node*&). At each node, a strcmp determines whether to descend left or right until a NULL slot is reached, where the new node is allocated in place — preserving sorted order with no additional passes.

BST Deletion — 3 Cases

Removal handles all three structural cases: leaf nodes are deleted directly; nodes with one child are bypassed; nodes with two children find the in-order successor by traversing to the leftmost node of the right subtree via the go_left() helper, which extracts and returns that node's data before unlinking it.

Remove by Source

remove_source() performs a full tree traversal — not a directed BST search — to find and remove every node whose ref field matches the given source. This demonstrates understanding of the distinction between key-based BST lookup and full-tree scans, applying the same three-case deletion logic at each match.

In-Order Display & Height

The display() function traverses left → visit → right, producing alphabetically sorted output automatically — a property of the BST structure itself, not a sorting step. The height() function computes tree depth recursively by comparing left and right subtree heights and returning the greater value plus one.

Code Samples

The recursive insert uses a node*& parameter — a reference to a pointer — so that assigning root = new node at any depth directly wires the new node into the tree's structure. There's no need to track a parent pointer separately; the reference handles the linkage automatically.

tree.cpp  —  insert_rec()

int trees::insert_rec(node *& root, definition to_add) {
  // base case: found an empty slot — allocate here
  if (!root) {
    root = new node;
    root->term.name     = to_add.name;
    root->term.def      = to_add.def;
    root->term.ref      = to_add.ref;
    root->term.citation = to_add.citation;
    root->left = root->right = NULL;
    return 1;
  }
  // go left if key is smaller, right otherwise
  if (strcmp(to_add.name, root->term.name) < 0)
    return insert_rec(root->left,  to_add);
  else
    return insert_rec(root->right, to_add);
}

tree.cpp  —  go_left() — in-order successor

// Find leftmost node: the in-order successor.
// Copy its data out, then unlink it from the tree.
int trees::go_left(node *& root, definition & users) {
  if (!root) return 1;

  if (!root->left) {
    // reached leftmost — save its data
    users.name     = root->term.name;
    users.def      = root->term.def;
    users.ref      = root->term.ref;
    users.citation = root->term.citation;

    // bypass this node (may have a right child)
    node *temp = root;
    root = root->right;
    temp->right = temp->left = NULL;
    delete temp;
    return 1;
  }
  return go_left(root->left, users);
}

When deleting a node with two children, the BST invariant must be preserved. go_left() finds the in-order successor — the smallest value in the right subtree — by recursing left until root->left is NULL. It copies that node's data into a definition& out-parameter, then unlinks and deletes the node — leaving the tree's sorted structure intact.

Workflow Graph — Adjacency List

Graph (Directed) Adjacency List Array of Vertices Linked List of Edges Pointer to Pointer OOP / Classes Dynamic Memory Linear Search


This project is a C++ graph application that models a directed workflow — a sequence of steps where each step can connect to one or more following steps. Vertices (steps) are stored in a heap-allocated array, and edges (connections) are stored as adjacency lists: each vertex holds a linked list of node pointers, each pointing directly to another vertex in the array.

The design keeps the graph compact and memory-efficient. Rather than copying vertex data into edge nodes, each node stores a raw vertex* — a pointer directly into the array — so displaying neighbors requires no lookup, just a single pointer dereference. The destructor fully traverses every adjacency list and deallocates all dynamic memory without leaks.

Key Implementation Details

Array of Vertices

The graph is backed by a dynamically allocated array of vertex structs, each holding a heap-allocated char* step name and a node* head to its adjacency list. Vertices are inserted sequentially into the next available slot, giving O(1) insertion when the position is known.

Pointer-to-Vertex Edges

Edge nodes don't copy vertex data — they store a vertex* adjacent pointing directly to the target vertex in the array. This is a deliberate design choice: it eliminates redundant storage and makes neighbor traversal a single pointer dereference rather than a search.

find_location()

find_location() performs a linear scan through the vertex array using strcmp to match by name and returns the index. Both insert_edge() and display_next() depend on this helper, keeping vertex lookup logic in one place and both callers clean.

Directed Edge Insert

insert_edge() prepends a new node to list[index1].head, setting its adjacent to &list[index2]. The edge is one-directional — only vertex1 gains a link to vertex2 — which makes this a directed graph where workflow order matters.

Code Samples

insert_edge() is the heart of the graph. It resolves both vertex names to array indices, allocates a new edge node, sets its adjacent to the address of the destination vertex in the array, and prepends it to the source vertex's adjacency list. The address-of operator (&list[index2]) is the key detail — it creates a live pointer into the array rather than a copy.

graph.cpp  —  insert_edge()

int table::insert_edge(char *vertex1, char *vertex2) {
  int index1 = find_location(vertex1);
  int index2 = find_location(vertex2);

  if (index1 < SIZE && index2 < SIZE) {
    node *temp    = new node;
    // point directly into the vertex array — no data copy
    temp->adjacent = &list[index2];
    temp->next     = list[index1].head;
    list[index1].head = temp;  // prepend to adjacency list
    return 1;
  }
  return 0; // one or both vertices not found
}

graph.cpp  —  display_next() + destructor

// one dereference to get the neighbor's name
int table::display_next(char *key_value) {
  int index = find_location(key_value);
  node *current = list[index].head;
  while (current) {
    cout << current->adjacent->step << endl;
    current = current->next;
  }

// destructor: free each adjacency list, then the array
table::~table() {
  for (int i = 0; i < SIZE; ++i) {
    delete [] list[i].step;
    node *cur = list[i].head;
    while (cur) {
      node *next = cur->next;
      cur->adjacent = NULL;
      delete cur;
      cur = next;
    }
  }
  delete [] list;
}

Because edge nodes hold a vertex* rather than copied data, displaying a neighbor's name is just current->adjacent->step — one pointer hop. The destructor mirrors this structure: it nulls each adjacent pointer before deleting the node (avoiding dangling pointer reads), then frees the vertex array itself — demonstrating careful, layered memory cleanup.
